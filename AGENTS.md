# AGENTS.md

## プロジェクト名

**HeyPJ!**

HeyPJ! は、Excel ベースの WBS ファイルで一般的に必要となる手作業での日付メンテナンスをなくすことを目的とした、プロジェクトスケジューリングおよび WBS 管理アプリケーションです。

核となる考え方はシンプルです。

> ユーザーが管理すべきは、タスク・工数・担当・依存関係・稼働状況である。
> スケジュールの計算は HeyPJ! が担う。

このアプリケーションは、タスクの開始日・終了日の手入力を可能な限り最小化すべきです。

---

# 1. プロダクトビジョン

従来の Excel による WBS 管理では、ユーザーは以下を繰り返し更新する必要があります。

- タスクの順序
- 開始日
- 終了日
- 担当者
- 見積工数
- 実績工数
- 依存関係
- 休日
- メンバーの稼働状況
- 会議のスケジュール

わずかな変更が、しばしば後続の多くの日付を不正確にしてしまいます。

HeyPJ! は、スケジュールを手作業で維持するデータではなく、計算された結果として扱うことで、この問題を解決すべきです。

基本原則は次のとおりです。

```text
プロジェクト開始日
+ タスク
+ 見積工数
+ タスクの依存関係
+ タスクの優先度 / 順序
+ 担当者
+ メンバーの稼働状況
+ 稼働カレンダー
=
自動生成されたスケジュール
```

ユーザーは主に入力を管理すべきです。

HeyPJ! は以下を自動的に計算すべきです。

- 計画開始日
- 計画終了日
- タスク期間
- リソース割り当て
- スケジュールの競合
- 後続スケジュールの変更
- プロジェクト完了日

---

# 2. 主要ターゲット

HeyPJ! は主に、WBS スタイルのワークフローを用いるソフトウェア開発プロジェクトを対象としています。

典型的なプロジェクトフェーズには以下が含まれます。

```text
要件定義
基本設計
詳細設計
開発
単体テスト
結合テスト
総合テスト
```

日本語での例:

```text
要件定義
基本設計
詳細設計
製造
単体テスト
結合テスト
総合テスト
```

アプリケーションはこれらのフェーズに限定されるべきではありませんが、これらは自然にサポートされるべきです。

---

# 3. コアドメインモデル

初期のドメインモデルは、以下の概念を中心に構成されるべきです。

```text
ワークスペース (Workspace)
プロジェクト (Project)
フェーズ (Phase)
タスク (Task)
メンバー (Member)
タスク依存関係 (Task Dependency)
稼働カレンダー (Work Calendar)
メンバー稼働状況 (Member Availability)
会議 / カレンダーブロック (Meeting / Calendar Block)
実績作業記録 (Actual Work Record)
```

モデルを不必要に複雑化しないでください。

汎用的な JSON フィールドよりも、明示的なドメイン概念を優先してください。

---

# 4. プロジェクト (Project)

プロジェクトは 1 つの開発イニシアチブを表します。

最小限のフィールド:

```text
id
name
description
start_date
timezone
default_workday_minutes
created_at
updated_at
```

例:

```text
プロジェクト名:
顧客管理システム フェーズ2

開始日:
2026-08-03

デフォルト稼働時間:
480分
```

プロジェクト開始日は、スケジューリングエンジンが使用する主要な入力の 1 つです。

---

# 5. フェーズ (Phase)

タスクはフェーズに属することができます。

典型的なフェーズ:

```text
要件定義
基本設計
詳細設計
開発
単体テスト
結合テスト
総合テスト
```

フェーズの順序は設定可能であるべきです。

すべてのプロジェクトが厳密なウォーターフォールの順序に従うことを前提とするビジネスロジックをハードコードしないでください。

プロジェクトはフェーズが重複することがあります。

例:

```text
フェーズ1 保守
フェーズ2 開発
```

これらは同時に進行する場合があります。

---

# 6. タスク (Task)

タスクは作業の基本単位です。

最小限のフィールド:

```text
id
project_id
phase_id
parent_task_id
title
description
estimated_minutes
actual_minutes
assignee_id
sort_order
status
planned_start_at
planned_end_at
actual_start_at
actual_end_at
created_at
updated_at
```

重要:

`planned_start_at` と `planned_end_at` は通常、計算された値です。

これらは、通常の手作業で維持されるフィールドとして扱われるべきではありません。

システムは後にスケジュールの手動オーバーライドをサポートする場合がありますが、自動計算がデフォルトです。

---

# 7. 見積工数

見積工数は内部的には分単位で保存されるべきです。

例:

```text
1人日 = 480分
0.5人日 = 240分
2時間 = 120分
```

UI では以下のように表示される場合があります。

```text
3人日
4時間
0.5人日
```

しかし、内部計算では一貫した単位を使用すべきです。

分を優先してください。

浮動小数点の日数計算をスケジューリングエンジンの基礎にしないでください。

---

# 8. 実績工数

タスクは実績工数をサポートしなければなりません。

最小限の要件:

```text
actual_minutes
```

後に、実績工数は個々の作業ログから導出される場合があります。

例:

```text
見積:
960分

実績:
1100分
```

見積工数と実績工数は分離されたままでなければなりません。

見積工数を実績工数で上書きしてはいけません。

---

# 9. タスクの並び替え

ユーザーはタスクを簡単に並び替えられるべきです。

UI はドラッグ&ドロップでの並び替えをサポートすべきです。

タスクの並び替えはスケジュールの再計算を引き起こす場合があります。

タスクの並び替え後に、ユーザーが開始日・終了日を手作業で書き直す必要があってはなりません。

プロダクトの核心的な価値は次のとおりです。

> 作業を並び替えれば、スケジュールが自動的に追従する。

ただし、表示順序と依存関係の順序は常に同一とは限りません。

以下を前提としないでください。

```text
行 N
は必ず
行 N + 1
より先に完了する
```

明示的なスケジューリングルールまたは依存関係がそれを要求する場合を除きます。

---

# 10. 依存関係

タスクは明示的な依存関係をサポートしなければなりません。

例:

```text
画面A 開発
    ↓
画面A 単体テスト
```

単体テストは、その開発タスクが完了するまで開始できません。

基本的な依存関係:

```text
predecessor_task_id
successor_task_id
dependency_type
```

最初の実装では、少なくとも以下をサポートすべきです。

```text
Finish-to-Start（終了-開始）
```

例:

```text
画面A 開発が完了してから
画面A 単体テストが開始する
```

アーキテクチャは、後で追加の依存関係タイプを許容できるようにすべきです。

将来考えられるタイプ:

```text
Finish-to-Start（終了-開始）
Start-to-Start（開始-開始）
Finish-to-Finish（終了-終了）
Start-to-Finish（開始-終了）
```

必要でない限り、すべての依存関係タイプを実装しないでください。

---

# 11. 完了依存関係

一部のタスクは、別のタスクが完了する前に開始できるものの、別のタスクが完了するまでは完了とみなせない場合があります。

例:

```text
画面A 基本設計
```

これは、API インターフェース設計がまだ進行中でも開始できる場合があります。

しかし:

```text
画面A 基本設計
```

は、以下が確定するまで完了できません。

```text
API A インターフェース設計
```

ドメインモデルは、完了制約を追加できるように設計されるべきです。

すべての依存関係が「開始できない」を意味するように強制しないでください。

将来の依存関係タイプは、次を表す場合があります。

```text
完了が別のタスクに依存する
```

この区別は、実際のソフトウェア開発プロジェクトにおいて重要です。

---

# 12. リソース制約

メンバー (Member) は人的リソースです。

人は、実際の利用可能なキャパシティを超える作業を割り当てられません。

1 人が無制限のタスクを同時にこなせると想定しないでください。

スケジューリングエンジンは、メンバーの利用可能な作業時間を考慮しなければなりません。

例:

```text
メンバー:
伊藤

1日あたりのキャパシティ:
480分
```

すでに 1 つのタスクに 300 分が割り当てられている場合、その日に別のタスクに使えるのは残り 180 分のみです。

---

# 13. 並行プロジェクトとフェーズ

メンバーは複数のプロジェクトやフェーズを同時に担当する場合があります。

例:

```text
フェーズ1 保守
フェーズ2 開発
```

両方が並行して進行する場合があります。

したがって、リソースの稼働状況は、単一のタスクやフェーズの内部だけでなく、メンバー全体で計算されなければなりません。

例:

```text
8時間労働日

フェーズ1 保守:
120分

フェーズ2 開発:
360分

合計:
480分
```

これは妥当です。

しかし:

```text
フェーズ1:
240分

フェーズ2:
360分

合計:
600分
```

これはキャパシティを超えており、検出されるか、自動的に再スケジュールされるべきです。

---

# 14. キャパシティ配分

システムは最終的にキャパシティ配分をサポートすべきです。

例:

```text
伊藤

フェーズ1 保守:
30%

フェーズ2 開発:
70%
```

8時間労働日の場合:

```text
フェーズ1:
144分

フェーズ2:
336分
```

この機能は最初の MVP で必ずしも必要ではありませんが、スケジューリングアーキテクチャがそれを不可能にすべきではありません。

---

# 15. 稼働カレンダー

スケジューリングは、すべての暦日が稼働日であると想定してはなりません。

プロジェクトまたは組織のカレンダーは、以下をサポートすべきです。

```text
週末
祝日
ゴールデンウィーク
夏季休暇
お盆
シルバーウィーク
年末年始休暇
会社独自の休日
```

スケジューリングエンジンは、利用不可の期間を自動的にスキップしなければなりません。

日本のプロジェクトでは、日本の祝日がサポートされるべきです。

会社によって稼働カレンダーが異なる場合があるため、休日の扱いは設定可能であるべきです。

---

# 16. メンバーの稼働状況

各メンバーは稼働カレンダーを持たなければなりません。

システムは以下をサポートすべきです。

```text
終日休暇
午前休
午後休
時間単位休暇
一時退席
研修
出張
社内イベント
その他の利用不可時間
```

稼働状況は、利用可能時間または利用不可時間の区間として表現されるのが望ましいです。

例:

```text
2026-08-05
終日休暇

利用可能:
0分
```

例:

```text
2026-08-06
午前休

利用可能:
240分
```

スケジューリングエンジンは、実際の利用可能な分数に対して計算すべきです。

---

# 17. 定例会議

定例会議は、タスク作業に使える時間を減少させます。

例:

```text
デイリースタンドアップ
11:00-11:30

顧客定例会議
火曜 14:00-15:00

社内設計レビュー
木曜 16:00-17:00
```

これらの期間は、スケジュール可能なタスクキャパシティを減少させなければなりません。

例:

```text
通常の作業キャパシティ:
480分

デイリースタンドアップ:
30分

利用可能なタスクキャパシティ:
450分
```

「1稼働日」が常にスケジュール可能な 8 時間を意味すると想定しないでください。

---

# 18. カレンダーブロック

利用不可の作業期間を表すために、CalendarBlock や AvailabilityBlock のような汎用的な概念を使用してください。

考えられるフィールド:

```text
id
member_id
project_id
title
start_at
end_at
type
recurrence_rule
created_at
updated_at
```

考えられるタイプ:

```text
meeting（会議）
leave（休暇）
holiday（休日）
training（研修）
business_trip（出張）
internal_event（社内イベント）
other（その他）
```

スケジューリングエンジンが主に関心を持つべきなのは、ある時間範囲がメンバーのキャパシティを消費するかどうかです。

---

# 19. スケジューリングエンジン

スケジューリングエンジンは、HeyPJ! において最も重要なドメインコンポーネントです。

スケジューリングロジックを UI コードから分離してください。

スケジューリング計算を React コンポーネント内に直接置かないでください。

専用のドメイン / サービスモジュールを優先してください。

例:

```text
SchedulingEngine
ScheduleCalculator
ResourceCalendar
DependencyResolver
```

スケジューリングエンジンは、スケジューリング入力を受け取り、計算されたスケジュール結果を生成します。

---

# 20. スケジューリング入力

スケジューラは少なくとも以下を考慮すべきです。

```text
プロジェクト開始日
タスクの見積工数
タスクの依存関係
タスクの順序 / 優先度
割り当てられたメンバー
メンバーの労働時間
メンバーの既存の割り当て
週末
祝日
会社の休日
メンバーの休暇
定例会議
臨時会議
その他のブロックされた時間
```

---

# 21. スケジューリング出力

スケジューラは以下を計算すべきです。

```text
planned_start_at
planned_end_at
日次の割り当て
リソース競合
依存関係の競合
プロジェクト完了日
```

有用な場合は、詳細な割り当て情報を保持してください。

例:

```text
タスクA

2026-08-03:
330分

2026-08-04:
450分

2026-08-05:
180分
```

これは、開始日と終了日のみを計算するよりも望ましいです。詳細な割り当てにより、スケジュールがどのように生成されたかを説明できるからです。

---

# 22. 基本スケジューリングルール

Finish-to-Start のタスク依存関係の場合:

```text
earliest_dependency_start
=
すべての先行タスクが完了した後の最初の利用可能な作業時間
```

タスクの実際の計画開始は、リソースの稼働状況も考慮すべきです。

概念的には:

```text
計画開始
=
以下のうち最も遅いもの:

- プロジェクトの稼働可能性
- 依存関係の稼働可能性
- 担当者の稼働可能性
```

その後、必要な作業がすべて割り当てられるまで、担当者の利用可能なキャパシティにわたってタスクの見積分数を消費します。

最終的に消費された時間が計画終了を決定します。

---

# 23. 再計算

スケジュールの再計算は、スケジューリング入力が変更されたときに行われるべきです。

例:

```text
タスクの並び替え
見積工数の変更
担当者の変更
依存関係の変更
メンバーの休暇追加
会議の追加
会議の削除
休日の追加
プロジェクト開始日の変更
```

最終的には、可能な場合には影響を受けるタスクのみを再計算すべきです。

最初の実装では、アルゴリズムが十分に高速であれば、プロジェクト全体の再計算でも許容されます。

正確性は、早すぎる最適化よりも重要です。

---

# 24. 影響分析

スケジュールを変更する操作が発生したとき、HeyPJ! は最終的にその影響を説明できるべきです。

例:

```text
伊藤が8月5日に休暇を追加。

影響を受けるタスク:
3件

画面A 開発
8月4日-6日
→
8月4日-7日

画面A 単体テスト
8月7日-8日
→
8月10日-11日
```

アーキテクチャは、将来の影響分析をサポートするのに十分な情報を保持すべきです。

---

# 25. 競合検出

システムは、無効または競合するスケジュールを検出すべきです。

例:

```text
リソースの過剰割り当て

循環タスク依存関係

先行タスク完了前にスケジュールされたタスク

終日休暇中に割り当てられたタスク

利用可能な作業分数を超えるタスク割り当て
```

循環依存関係は、決して黙って受け入れられてはなりません。

無効な依存関係の例:

```text
タスクA
↓
タスクB
↓
タスクC
↓
タスクA
```

明確なバリデーションエラーを返してください。

# 26. WBS UI

メインの WBS インターフェースは、Jira よりも Excel に近い感覚であるべきです。

ユーザーは行と列で素早く作業できるべきです。

望ましい挙動:

```text
インラインセル編集
キーボードナビゲーション
Tab ナビゲーション
Enter ナビゲーション
コピー&ペースト
一括行操作

展開可能な階層
```

単純な編集のたびにモーダルを開かせることは避けてください。

# 27. WBS 列

初期の WBS 列には以下を含めるべきです。

```text
タスク
フェーズ
見積工数
実績工数
担当者
計画開始
計画終了
実績開始
実績終了

```

追加候補の列:


そうでない場合は、それらが計算値であることを明確にしてください。

# 28. フィルタリングとビュー

大規模な WBS では、すべてのタスクを一度に見るのは非効率です。

ユーザーは、関心のあるタスクだけに素早く絞り込めるべきです。

少なくとも以下でフィルタリングできるべきです。

```text
担当者
フェーズ
ステータス
```

特に担当者でのフィルタリングは重要です。

例:

```text
担当者:
伊藤

→ 伊藤に割り当てられたタスクのみ表示
```

これにより、各メンバーが「自分の作業」を確認したり、マネージャーが特定メンバーの負荷を把握したりできます。

重要な原則:

フィルタリングは表示のみに影響すべきであり、スケジューリング計算には影響してはなりません。

```text
フィルタで一部のタスクを隠しても、
スケジュールは全タスクを考慮して計算されたままである。
```

フィルタされた結果だけを使ってスケジュールを再計算しないでください。

フィルタリングは、WBS テーブルとガントビューの両方に一貫して適用されるべきです。

複数の担当者や複数フェーズの同時選択を、将来的にサポートできるようにしてください。

---

# 29. 親タスク

タスクは階層を形成できます。

例:

```text
基本設計
├── 画面A
├── 画面B
└── API A
```

階層には以下を使用してください。

```text
parent_task_id
sort_order
```

以下のような、手動で保存された WBS 番号に依存しないでください。

```text
1
1.1
1.1.1
```

これらは、現在の階層と順序から生成されるべきです。

これにより、ドラッグ&ドロップ操作後の番号振り直しの問題を防げます。

# 36. 技術スタックとインフラ（Cloudflare 前提）

HeyPJ! は Cloudflare プラットフォーム上に構築することを前提とします。

基本構成:

```text
フロントエンド（WBS / ガント UI）
  React SPA
  → Cloudflare Workers Static Assets で配信

バックエンド API
  → Cloudflare Workers

データベース
  → Cloudflare D1（SQLite）

プロジェクト単位の調整・再計算・同時編集
  → Durable Objects（プロジェクトごとに1インスタンス）

セッション / キャッシュ
  → Workers KV

ファイル（Excel 入出力など）
  → R2
```

## データベース: D1

ドメインモデル（プロジェクト・フェーズ・タスク・メンバー・依存関係・カレンダー）はリレーショナルであり、D1（SQLite）が自然に適合します。

見積・実績工数は整数の分で保存するため、SQLite の整数型と相性が良いです。

浮動小数点で日数を保持しないでください（「見積工数」「スケジューリングコードの要件」を参照）。

スキーマ変更は wrangler の D1 マイグレーションで管理してください。

## スケジューリングエンジンと Durable Objects

スケジューリングエンジンは、あくまで純粋で決定論的なドメインモジュールのままでなければなりません（「アーキテクチャ原則」「スケジューリングコードの要件」を参照）。

Durable Objects は、そのエンジンを「呼び出す」調整レイヤーとして使用します。

スケジューリングのロジックそのものを Durable Object の中に埋め込まないでください。

プロジェクトごとに 1 つの Durable Object を割り当てることで、次を得られます。

```text
再計算の直列化（競合状態が発生しない）
プロジェクトスケジュールの一貫した信頼できる情報源
WebSocket による WBS のリアルタイム同時編集
alarm によるスケジュール処理の予約
```

スケジューリング入力が変わったとき（「再計算」を参照）は、その変更をプロジェクトの Durable Object を通して適用し、再計算を直列に実行してください。

Durable Objects 利用上の注意:

```text
1 インスタンスあたり約 1,000 req/s が上限（プロジェクト単位なら十分）
alarm は 1 インスタンスにつき 1 つ。予約はキューパターンで束ねる
ハイバネーションで in-memory 状態は失われる
確定データは D1 / DO ストレージへ永続化する
```

## タイムゾーン

Workers は UTC で実行されます。

サーバーやランタイムのタイムゾーンに依存せず、プロジェクトに設定されたタイムゾーン（既定 Asia/Tokyo）で日付計算を行ってください（「タイムゾーン」を参照）。

## Cloudflare 非依存の維持

スケジューリングのコアドメインは、Cloudflare 固有の API に依存すべきではありません。

D1・Durable Objects・KV・R2 へのアクセスは、リポジトリ / インフラ層の背後に隠してください。

これにより、スケジューリングエンジンは Cloudflare ランタイム外でも単体テスト可能なままになります。

外部プラットフォーム（Cloudflare）をインフラとして採用することは、「外部連携」で述べた早計なサードパーティ結合とは別物です。

スケジューリングの信頼できる情報源は、引き続き HeyPJ! 自身が保持します。

# 37. スケジューリングコードの要件

- 決定論的である
- 独立してテスト可能である
- 隠れたグローバル状態を避ける
- 明示的な入力を使用する
- 明示的な出力を返す
- タイムゾーン依存の暗黙的な挙動を避ける
- 浮動小数点の時間演算を避ける

同じ入力が与えられれば、スケジューリングエンジンは常に同じスケジュールを返さなければなりません。

# 43. 中核となるユーザー体験

```text
依存関係の変更
```

ユーザーが後続のすべての日付を手作業で修復する必要があってはなりません。

HeyPJ! がそれらを再計算すべきです。

# 44. 重要なプロダクトステートメント

このアプリケーションは、常にこの概念を保つべきです。

> **ユーザーは作業を管理する。HeyPJ! は日付を管理する。**

日本語:

> **人が管理するのは、作業・工数・担当・依存関係。
> 日付の管理はHeyPJ!に任せる。**

もう 1 つの適切なプロダクトステートメント:

> **WBSの日付を、人間がメンテナンスしない。**

これが中心的なプロダクト哲学です。

# 46. AI コーディングエージェントへの指示

このリポジトリを変更する際は:

1. アーキテクチャの変更を行う前に、この AGENTS.md を読むこと。
2. スケジューリングファーストのプロダクト哲学を維持すること。
3. 明確な要件なしに外部サービスへの依存を導入しないこと。
4. UI コンポーネント内にスケジューリングルールを重複させないこと。
5. スケジューリングの挙動が変わるときは、常にスケジューリングテストを追加または更新すること。
6. 一般化された抽象よりも、シンプルなドメインモデルを優先すること。
7. 安全に導出できる値を手作業で永続化しないこと。
8. タスクの行順序が自動的に依存関係の順序を意味すると決して想定しないこと。
9. すべての稼働日が同じ利用可能キャパシティを持つと決して想定しないこと。
10. スケジューリングロジックを変更する際は、常に週末・休日・会議・休暇・リソース競合を考慮すること。
11. 計算された計画日を、スケジューリングエンジンの出力として扱うこと。
12. すべての日付/時刻計算をタイムゾーン対応に保つこと。
13. 内部の工数計算には整数の分を使用すること。
14. スケジュール計算の前に依存関係の循環を検出すること。
15. 複数のフェーズやプロジェクトにまたがって割り当てられたメンバーでも変更が機能することを保証すること。
