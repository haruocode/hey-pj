# HeyPJ!

**WBSの日付を、人間がメンテナンスしない。**

HeyPJ! は、Excel ベースの WBS で発生しがちな「手作業での日付メンテナンス」をなくすことを目的とした、プロジェクトスケジューリング / WBS 管理アプリケーションです。

> 人が管理するのは、作業・工数・担当・依存関係。
> 日付の管理は HeyPJ! に任せる。

---

## コンセプト

従来の Excel による WBS 管理では、わずかな変更のたびに開始日・終了日・後続タスクを手作業で修正する必要があります。HeyPJ! は、スケジュールを **手作業で維持するデータではなく、計算された結果** として扱います。

```text
プロジェクト開始日
+ タスク
+ 見積工数
+ タスクの依存関係
+ タスクの優先度 / 順序
+ 担当者
+ メンバーの稼働状況
+ 稼働カレンダー
=
自動生成されたスケジュール
```

ユーザーは入力（タスク・工数・担当・依存関係・稼働状況）を管理し、以下は HeyPJ! が自動で計算します。

- 計画開始日 / 計画終了日
- タスク期間
- リソース割り当て
- スケジュール / 依存関係の競合
- プロジェクト完了日

---

## 主な特徴

- **スケジューリングエンジン** — 決定論的で独立してテスト可能なドメインコンポーネント。同じ入力からは常に同じスケジュールを生成します。
- **リソース制約** — メンバーのキャパシティ（分単位）を考慮し、過剰割り当てを検出します。複数プロジェクト / フェーズをまたぐ稼働も計算対象です。
- **稼働カレンダー** — 週末・日本の祝日・会社独自の休日・メンバーの休暇（終日 / 半日）・定例会議を反映します。
- **依存関係** — Finish-to-Start を基本にサポート。循環依存はバリデーションエラーとして検出します。
- **Excel ライクな WBS UI** — インライン編集、キーボード操作、ドラッグ&ドロップ並び替え、階層表示。
- **自動再計算** — タスクの並び替え・工数変更・休暇追加などの入力変更に、スケジュールが自動追従します。

## 技術スタック（Cloudflare 前提）

| レイヤー | 採用技術 |
| --- | --- |
| フロントエンド（WBS / ガント UI） | React SPA（Cloudflare Workers Static Assets で配信） |
| バックエンド API | Cloudflare Workers |
| データベース | Cloudflare D1（SQLite） |
| プロジェクト単位の調整・再計算・同時編集 | Durable Objects（プロジェクトごとに 1 インスタンス） |
| セッション / キャッシュ | Workers KV |
| ファイル（Excel 入出力など） | R2 |

- 工数は **整数の分** で保存します（浮動小数点の日数計算は使用しません）。
- スケジューリングのコアドメインは Cloudflare 固有の API に依存させず、リポジトリ / インフラ層の背後に隠します（ランタイム外でも単体テスト可能に保つため）。
- 日付計算はサーバー / ブラウザのタイムゾーンに依存せず、プロジェクト設定のタイムゾーン（既定: `Asia/Tokyo`）で行います。

## アーキテクチャ

スケジューリングドメインをフレームワーク固有の実装から分離します。

```text
src/
  domain/          # project / task / member / calendar / scheduling
  application/     # create-task / reorder-task / assign-member / recalculate-schedule
  infrastructure/  # database / repositories
  features/        # wbs / gantt / members / calendars
```

スケジューリングロジックを React コンポーネントや Durable Object の中に埋め込まないでください。Durable Object はエンジンを「呼び出す」調整レイヤーとして使用します。

## プロダクト原則

機能を追加すべきか迷ったら、この問いに立ち返ります。

> これは手作業の WBS スケジュールメンテナンスを減らすか?

減らさないなら、それはおそらく HeyPJ! のコア機能ではありません。

開発上の意思決定が衝突する場合の優先順位:

1. スケジューリングの正確性
2. データ整合性
3. WBS 編集のしやすさ
4. 明確なドメインモデル
5. テスト可能性
6. パフォーマンス
7. 視覚的な洗練
8. 追加機能

## 開発

```bash
pnpm install
pnpm test         # ドメイン/アプリ層のユニットテスト（node 環境）
pnpm typecheck
pnpm build        # クライアント SPA + Workers バンドル

# ローカル D1（初回・スキーマ変更時）
pnpm build
npx wrangler d1 migrations apply DB --local

# ローカル起動（Workers + D1 + Durable Object をローカル実行）
npx wrangler dev
```

API（プロジェクト単位で Durable Object が調整）:

| メソッド | パス | 用途 |
| --- | --- | --- |
| `GET` | `/api/projects/:id/schedule` | 現在のスケジュールを取得 |
| `POST` | `/api/projects/:id/tasks` | タスク作成 → 再計算 |
| `POST` | `/api/projects/:id/assign` | 担当者変更 → 再計算 |
| `POST` | `/api/projects/:id/reorder` | 並び替え → 再計算 |
| `POST` | `/api/projects/:id/recalculate` | 明示的に再計算 |

> `wrangler` の対応範囲に合わせ、`compatibility_date` は現状 `2024-11-01`。デプロイ時は
> `wrangler d1 create hey-pj` で得た ID を `wrangler.jsonc` の `database_id` に設定します。
